# Tent

Podman-based CLI tool for running local development dependencies (databases, caches, message brokers, mail catchers) as pre-configured containers. Linux only, rootless Podman.

## Build and run

```bash
make build          # builds to bin/tent
make install        # installs via go install
go build ./...      # compile check
go test ./...       # run tests
go vet ./...        # static analysis
make check          # fmt + vet + lint + test
```

Version is derived from git tags via `git describe --tags`. The Makefile injects it through ldflags into `cmd.version`. GoReleaser does the same for releases.

## Project structure

```
main.go                 Entry point, calls cmd.Execute()
cmd/                    Cobra CLI commands (root, start, stop, list, services, version)
services/               One file per service, each exports a types.Service variable
store/                  Service registry — maps names to service definitions
runtime/                Podman v5 bindings wrapper (connect, create, start, stop, remove, list)
types/                  Shared structs: Service, EnvVar, PortMapping, VolumeMount
.github/workflows/      CI (push/PR) and Release (tag push) GitHub Actions
.goreleaser.yaml        GoReleaser config for cross-platform builds
```

## How services work

Each service is a `types.Service` struct defined as a package-level variable in `services/`. The struct holds: image name, tag, port mappings, environment variables, volumes, optional command, and optional insecure mode config.

Services are registered by name in the `store/services.go` registry map. The `store.GetService()` function returns deep copies so callers can mutate without affecting the registry.

To add a new service:
1. Create `services/<name>.go` with a `var <Name> types.Service` following existing patterns
2. Add the entry to the `registry` map in `store/services.go`
3. That's it — the service is automatically available in `tent start`, `tent stop`, `tent services`, and tab completion

## Key patterns

- **ContainerEngine interface**: `runtime.ContainerEngine` abstracts container operations. `*Runtime` implements it. Command functions accept the interface so tests can substitute a mock
- **Extracted command logic**: `startServices()`, `stopServices()`, `listContainers()` hold the actual logic; Cobra `Run` closures just call `Connect()` then delegate. This keeps Podman coupling at the edges
- **Deep copy on read**: `GetService()` copies all slices before returning to prevent mutation of registry state
- **Insecure mode**: Services with `InsecureEnv` set support `--insecure`. `nil` InsecureEnv means unsupported. Empty slice `[]EnvVar{}` means "drop all auth env vars" (used by MongoDB, SurrealDB)
- **Container naming**: `tent-{service}-{tag}-{hostport}` — generated by `Service.ContainerName()`
- **Container labels**: All tent containers get `tent.managed=true` and `tent.service=<name>` labels for filtering
- **Interactive prompts**: Without `--default`/`-d`, the start command prompts for tag, ports, mutable env vars, and volume names via fmt.Scanln
- **Restart policies**: Passed through to Podman's container spec directly

## Dependencies

- `github.com/containers/podman/v5` — Podman API bindings
- `github.com/containers/common` — Podman network types
- `github.com/spf13/cobra` — CLI framework
- `github.com/spf13/viper` — Config file support (mostly scaffolding, not heavily used yet)

All builds are static (`CGO_ENABLED=0`) with the `containers_image_openpgp` build tag, which uses pure Go PGP instead of the C `libgpgme`. No C compiler or system libraries needed. The Makefile, GoReleaser, and CI all use this same configuration.

## Releasing

Push a semver tag to trigger the release workflow:

```bash
git tag v0.1.0
git push origin v0.1.0
```

GoReleaser builds linux/darwin binaries for amd64/arm64, creates checksums, and publishes a GitHub Release.
